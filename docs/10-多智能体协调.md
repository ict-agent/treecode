# 多智能体协调

OpenHarness 实现了完整的多 Agent 协作系统，包括 Swarm 执行后端、后台任务管理和团队协调。

> 对应源码：`src/openharness/swarm/` + `src/openharness/coordinator/` + `src/openharness/tasks/`

---

## 整体架构

```
Leader Agent（主 agent）
    │
    ├── agent_tool.py ──► BackendRegistry（自动选择执行后端）
    │                       ├── InProcessBackend  （同进程 asyncio Task）
    │                       ├── SubprocessBackend （子进程，复用 TaskManager）
    │                       └── PaneBackend       （tmux/iTerm2 可视化面板）
    │
    ├── TeammateMailbox ──► 文件信箱（JSON 消息队列）
    │                       ~/.openharness/teams/<team>/agents/<id>/inbox/
    │
    ├── PermissionSync ──► 权限代理（worker → leader）
    │                       pending/<id>.json → resolved/<id>.json
    │
    ├── TeamLifecycleManager ──► 团队持久化
    │                             ~/.openharness/teams/<name>/team.json
    │
    └── WorktreeManager ──► Git worktree 隔离
                              每个 teammate 独立工作树
```

## Swarm 执行后端

> 源码：`src/openharness/swarm/`（~4,800 行）

### 核心类型

> 源码：[`swarm/types.py`](../src/openharness/swarm/types.py)

```python
BackendType = Literal["subprocess", "in_process", "tmux", "iterm2"]

@dataclass
class TeammateSpawnConfig:
    name: str                    # teammate 名称
    team: str                    # 所属团队
    prompt: str                  # 初始 prompt
    cwd: str                     # 工作目录
    parent_session_id: str       # 父会话 ID
    model: str | None = None     # 模型覆盖
    system_prompt: str | None = None
    permissions: list[str] = field(default_factory=list)
    worktree_path: str | None = None  # Git worktree 路径
    # ...更多配置

@dataclass
class SpawnResult:
    task_id: str                 # 任务 ID
    agent_id: str                # agentName@teamName
    backend_type: BackendType    # 使用的后端
    success: bool = True
    error: str | None = None

class TeammateExecutor(Protocol):
    type: BackendType
    def is_available(self) -> bool: ...
    async def spawn(self, config: TeammateSpawnConfig) -> SpawnResult: ...
    async def send_message(self, agent_id: str, message: TeammateMessage) -> None: ...
    async def shutdown(self, agent_id: str, *, force: bool = False) -> bool: ...
```

### BackendRegistry — 自动检测与分发

> 源码：[`swarm/registry.py`](../src/openharness/swarm/registry.py)

```python
registry = get_backend_registry()
executor = registry.get_executor("in_process")  # 指定后端
executor = registry.get_executor()               # 自动检测
```

自动检测优先级：
1. 运行在 tmux 中？→ tmux backend
2. 运行在 iTerm2 中？→ iTerm2 backend
3. 否则 → in_process backend（默认）

### InProcessBackend — 同进程执行

> 源码：[`swarm/in_process.py`](../src/openharness/swarm/in_process.py)

用 `contextvars` 实现 per-teammate 上下文隔离（Python 版的 AsyncLocalStorage）：

- `TeammateAbortController`：双信号取消（graceful cancel + force kill）
- `TeammateContext`：身份、abort controller、运行统计
- `start_in_process_teammate`：启动查询引擎、驱动循环、清理退出
- 每个 teammate 是一个 `asyncio.Task`，不额外开进程

### SubprocessBackend — 子进程执行

> 源码：[`swarm/subprocess_backend.py`](../src/openharness/swarm/subprocess_backend.py)

复用现有 `BackgroundTaskManager`，通过 stdin/stdout 通信。构建命令时继承父进程的 CLI 参数和环境变量。

## 消息信箱

> 源码：[`swarm/mailbox.py`](../src/openharness/swarm/mailbox.py)

基于文件的异步消息队列，每条消息是一个 JSON 文件：

```
~/.openharness/teams/<team>/agents/<agent_id>/inbox/<timestamp>_<msg_id>.json
```

原子写入（`.tmp` + `os.rename`），`fcntl` 文件锁防止并发读写冲突。

```python
@dataclass
class MailboxMessage:
    id: str
    type: MessageType   # "user_message", "permission_request", "shutdown", ...
    sender: str
    recipient: str
    payload: dict[str, Any]
    timestamp: float
```

## 权限同步

> 源码：[`swarm/permission_sync.py`](../src/openharness/swarm/permission_sync.py)

Worker 执行敏感操作时需要向 Leader 请求批准。支持两种方式：

**文件方式**：
```
~/.openharness/teams/<team>/permissions/pending/<id>.json   → Worker 写入
~/.openharness/teams/<team>/permissions/resolved/<id>.json  → Leader 批准/拒绝
```

**信箱方式**：通过 `TeammateMailbox` 的 `permission_request` / `permission_response` 消息类型。

## Git Worktree 隔离

> 源码：[`swarm/worktree.py`](../src/openharness/swarm/worktree.py)

为每个 teammate 分配独立的 git worktree，避免多 agent 同时编辑同一文件：

```python
class WorktreeManager:
    async def create(self, slug: str) -> WorktreeInfo
    async def remove(self, slug: str) -> None
    async def list_worktrees(self) -> list[WorktreeInfo]
```

Slug 验证规则：最长 64 字符，只允许字母数字和 `._-`，禁止路径穿越。

## 工具集成

`agent_tool.py` 现在通过 swarm BackendRegistry 分发：

```python
# agent_tool.py 核心流程
agent_def = get_agent_definition(arguments.subagent_type)  # 可选
registry = get_backend_registry()
```

| 工具 | 功能 |
|------|------|
| `agent` | 生成 teammate（通过 swarm 后端） |

| `task_create` / `task_get` / `task_list` / `task_update` / `task_stop` / `task_output` | 后台任务管理 |

## 设计要点

1. **后端抽象**：`TeammateExecutor` 协议统一了 in-process、subprocess、tmux 三种执行方式
2. **进程内优先**：默认使用 `InProcessBackend`，避免子进程开销
3. **文件信箱**：原子 JSON 文件 + 文件锁，无需外部消息中间件
4. **权限代理**：Worker 不直接决定权限，通过 Leader 审批
5. **Worktree 隔离**：多 agent 并行编辑时避免文件冲突
6. **向后兼容**：`BackgroundTaskManager` 和内存 `TeamRegistry` 保留，作为基础层

## 当前局限与未来方向

### 当前局限

- tmux/iTerm2 PaneBackend 的 Python 实现尚未完成（类型定义已就绪）
- `InProcessBackend` 依赖 `contextvars` 隔离，深度嵌套的 agent 可能需要更完善的上下文传播
- 信箱轮询机制是定时检查，尚无事件驱动的推送通知

### 未来方向

- 完成 tmux/iTerm2 可视化面板后端
- Agent 间更丰富的协议（beyond 纯文本消息）
- 跨主机的 agent 协调
- [ClawTeam](https://github.com/HKUDS/ClawTeam) 集成
