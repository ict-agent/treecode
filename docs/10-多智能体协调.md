# 多智能体协调

TreeCode 实现了完整的多 Agent 协作系统，包括 Swarm 执行后端、后台任务管理和团队协调。

> 对应源码：`src/treecode/swarm/` + `src/treecode/coordinator/` + `src/treecode/tasks/`

---

## 整体架构

```
Leader Agent（主 agent）
    │
    ├── agent_tool.py ──► BackendRegistry（自动选择执行后端）
    │                       ├── InProcessBackend  （同进程 asyncio Task）
    │                       ├── SubprocessBackend （子进程，复用 TaskManager）
    │                       └── PaneBackend       （tmux/iTerm2 可视化面板）
    │
    ├── TeammateMailbox ──► 文件信箱（JSON 消息队列）
    │                       ~/.treecode/teams/<team>/agents/<id>/inbox/
    │
    ├── PermissionSync ──► 权限代理（worker → leader）
    │                       pending/<id>.json → resolved/<id>.json
    │
    ├── TeamLifecycleManager ──► 团队持久化
    │                             ~/.treecode/teams/<name>/team.json
    │
    └── WorktreeManager ──► Git worktree 隔离
                              每个 teammate 独立工作树
```

## Swarm 执行后端

### 核心类型

> 源码：[`swarm/types.py`](../src/treecode/swarm/types.py)

```python
BackendType = Literal["subprocess", "in_process", "tmux", "iterm2"]

@dataclass
class TeammateSpawnConfig:
    name: str                    # teammate 名称
    team: str                    # 所属团队
    prompt: str                  # 初始 prompt
    cwd: str                     # 工作目录
    parent_session_id: str       # 父会话 ID
    model: str | None = None     # 模型覆盖
    system_prompt: str | None = None
    permissions: list[str] = field(default_factory=list)
    worktree_path: str | None = None  # Git worktree 路径
    # ...更多配置

@dataclass
class SpawnResult:
    task_id: str                 # 任务 ID
    agent_id: str                # agentName@teamName
    backend_type: BackendType    # 使用的后端
    success: bool = True
    error: str | None = None

class TeammateExecutor(Protocol):
    type: BackendType
    def is_available(self) -> bool: ...
    async def spawn(self, config: TeammateSpawnConfig) -> SpawnResult: ...
    async def send_message(self, agent_id: str, message: TeammateMessage) -> None: ...
    async def shutdown(self, agent_id: str, *, force: bool = False) -> bool: ...
```

### BackendRegistry — 自动检测与分发

> 源码：[`swarm/registry.py`](../src/treecode/swarm/registry.py)

```python
registry = get_backend_registry()
executor = registry.get_executor("in_process")  # 指定后端
executor = registry.get_executor()               # 自动检测
```

自动检测优先级：
1. 运行在 tmux 中？→ tmux backend
2. 运行在 iTerm2 中？→ iTerm2 backend
3. 否则 → in_process backend（默认）

### InProcessBackend — 同进程执行

> 源码：[`swarm/in_process.py`](../src/treecode/swarm/in_process.py)

用 `contextvars` 实现 per-teammate 上下文隔离（Python 版的 AsyncLocalStorage）：

- `TeammateAbortController`：双信号取消（graceful cancel + force kill）
- `TeammateContext`：身份、abort controller、运行统计
- `start_in_process_teammate`：启动查询引擎、驱动循环、清理退出
- 每个 teammate 是一个 `asyncio.Task`，不额外开进程

### SubprocessBackend — 子进程执行

> 源码：[`swarm/subprocess_backend.py`](../src/treecode/swarm/subprocess_backend.py)

复用现有 `BackgroundTaskManager`，通过 stdin/stdout 通信。构建命令时继承父进程的 CLI 参数和环境变量。

## 后台任务管理器

> 源码：[`tasks/manager.py`](../src/treecode/tasks/manager.py)

`BackgroundTaskManager` 仍然负责子进程级别的任务管理，被 `SubprocessBackend` 使用。

```python
TaskType = Literal["local_bash", "local_agent", "remote_agent", "in_process_teammate"]
TaskStatus = Literal["pending", "running", "completed", "failed", "killed"]
```

任务输出写入 `~/.treecode/data/tasks/<task_id>.log`。

### 拓扑 SoT vs 任务 SoT

Swarm 与 task 虽然都落盘，但它们是两套不同的事实来源，不能混用：

- **Swarm 拓扑 SoT**
  - `~/.treecode/data/swarm/events.jsonl`
  - 记录 `agent_spawned`、`agent_reparented`、`agent_finished` 等结构与生命周期事件
  - 任意时刻的全局树由 `EventStore + SwarmProjection + RuntimeGraph` 物化得到
- **Task SoT**
  - `~/.treecode/data/tasks/<task_id>.json` + `*.log`
  - 记录后台任务元数据、输出和运行状态
  - `task_get` / `task_list` 可以合并磁盘记录做跨进程可见，但这不等价于 swarm 树

一句话区分：**谁是谁的父子关系，看 swarm 事件流；某个后台进程现在是否还活着、输出了什么，看 task record 和 task log。**

### 共享数据目录要求

无论 leader、子 agent 子进程还是 Web console backend，若要看到同一棵 swarm 树，必须共享同一 `TREECODE_DATA_DIR`（或都使用默认 home data 目录）。

如果某个子进程指向了不同的数据目录：

- 它会把 `events.jsonl` 写到另一套路径
- 其它进程重放到的是另一条事件流
- 结果是「拓扑分裂」，而不是投影算法出错

因此多进程一致性首先是**共享数据目录约定**，然后才是事件文件写入策略。

## 设计要点

1. **后端抽象**：`TeammateExecutor` 协议统一了 in-process、subprocess、tmux 三种执行方式
2. **进程内优先**：默认使用 `InProcessBackend`，避免子进程开销
3. **文件信箱**：原子 JSON 文件 + 文件锁，无需外部消息中间件
4. **权限代理**：Worker 不直接决定权限，通过 Leader 审批
5. **Worktree 隔离**：多 agent 并行编辑时避免文件冲突
6. **向后兼容**：`BackgroundTaskManager` 和内存 `TeamRegistry` 保留，作为基础层

### live 运行时语义

当前浏览器控制台中的 `live` 模式不再只是“把当前还能看到的后台 agent 平铺出来”，而是有一个稳定的树根：

- 首次进入 `live` source 时，控制台会自动确保存在 `main@default`
- 从控制台直接创建的 `live` agent，若没有显式指定 parent，会默认挂到 `main@default`
- live agent 自己继续创建 subagent 时，则沿已有 `parent_agent_id / root_agent_id / lineage_path` 挂树

这让 live 树结构与 deterministic `scenario` 模式在心智模型上保持一致：

```text
main@default
├── worker-a@default
└── worker-b@default
```

### `active_source` vs `topology_view`

Web console 里有两个容易混淆但应当分开的维度：

- `active_source`
  - 选择事件源：`live` 或 `scenario`
  - `live` 读取真实运行中的 event store / context / task 状态
  - `scenario` 读取 deterministic synthetic 场景
- `topology_view`
  - 选择如何看当前 source 的拓扑
  - `raw_events`：纯事件重放得到的树，保留历史结点
  - `live`：在 `raw_events` 基础上，再应用 `reconcile_live_runtime` 过滤，仅保留当前仍被运行时视为活跃的 agent 及其祖先

因此：

- `active_source=live + topology_view=live` 表示“真实运行时当前活着的树”
- `active_source=live + topology_view=raw_events` 表示“真实运行时到目前为止写过的完整历史树”
- `active_source=scenario` 下通常不需要额外 live reconcile，但仍沿用同一 snapshot 协议字段，避免前后端语义分叉

### live 恢复与实时推送

为保证浏览器控制台“真实可用”，当前实现还补了两层能力：

- **持久 live agent 恢复**
  - `SubprocessBackend` 会在内存映射缺失时，尝试根据持久化的 `agent_spawned.task_id` 与 task record 恢复 `agent_id -> task_id`
  - `BackgroundTaskManager` 也支持在 manager 重启后重新加载 task record
- **后台自动推送 snapshot**
  - 之前只有命令执行时才广播 snapshot
  - 现在 live agent 在后台产生 `assistant_message` / `tool_*` 等事件后，WebSocket server 会自动向已连接前端推送新 snapshot

因此对 live agent 发送 follow-up 时，页面会先看到 `incoming debugger...`，随后自动看到新的 `assistant` 回复，无需手动刷新。

## 当前局限与未来方向

### 当前局限

- tmux/iTerm2 PaneBackend 的 Python 实现尚未完成（类型定义已就绪）
- `InProcessBackend` 依赖 `contextvars` 隔离，深度嵌套的 agent 可能需要更完善的上下文传播
- 信箱轮询机制是定时检查，尚无事件驱动的推送通知
- Web console 仍在迭代中，交互与视觉层仍有继续产品化空间

### 未来方向

- 完成 tmux/iTerm2 可视化面板后端
- Agent 间更丰富的协议（beyond 纯文本消息）
- 跨主机的 agent 协调
- [ClawTeam](https://github.com/HKUDS/ClawTeam) 集成
